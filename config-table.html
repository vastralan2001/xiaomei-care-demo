A/B 实验设计

分桶策略 · 核心指标 · 观测方案
维度 说明
分桶方式 (userId.hashCode() & 0x7FFFFFFF) % 100
bucket 0-49 → 对照组 G1  |  bucket 50-99 → 实验组 G3
Phase 1 只分 G1/G3，Phase 2 扩展为 G1(对照)/G2(仅发券)/G3(仅挂店)/G4(挂店+发券)
场景共用分桶 4 个场景共用同一个 G1/G3 分桶，不按场景各自 AB。
原因：单场景样本量不够拆，先验证"关怀+挂店+券"整体模式的增量效果。
对照组行为 正常推 3 家店 → 结束。不触发 searchHealthPoi，不注入 health_context，不出券。
实验组行为 正常推 3 家店 → 并行搜关怀店 → 注入 health_context → LLM 追加关怀段 + 挂店 + 红包。
超时降级 searchHealthPoi 3 秒超时，失败则实验组也降级为对照组体验（不出关怀段）。
核心指标
  • 转化率：关怀店点击率、关怀店下单率、红包领取率
  • 客单价：实验组 vs 对照组平均客单价变化
  • 留存：次日 / 7日回访率差异
  • 体验：对话满意度评分、负反馈率（"不需要推荐"等）

需新增能力 / 数据源

4 场景涉及的外部依赖和改造点

P0 场景路由 sceneType

在 RunnerContext 新增 healthSceneType 字段（unhealthy / weather / latenight / mood），用于动态选取不同的 query、template、coupon 配置。HealthRecommendService 返回时设置。

P0 Lion 配置 Map 化

将 extra.search.query、chat.context.template、coupon.mock.text 从 String 改为 Map<String, String>，key=sceneType，value=场景配置。

P0 时间条件判断

在 isTargetUser() 中新增时间分支：工作日 22:00-02:00 → 命中深夜场景。仅需 LocalDateTime + DayOfWeek，无外部依赖。节假日 API 可选。

P1 天气 API 接入

接入实时天气数据源（推荐：彩云天气 / 和风天气 / 美团内部气象服务）。按用户定位城市获取当日温差、降水概率。需增加缓存避免 API 限频。

P1 NLU 情绪检测

Phase 1 用关键词规则兜底（"随便""算了""无所谓""都行""不知道"等）。Phase 2 接入情绪分类模型（对话上下文 → 情绪标签）。需与 NLU 团队对接。

P1 券真实发放

Phase 2 对接 callRedEnvelopSender，使用真实 activityId（参照"开工红包"子 agent 发券链路）。需配置各场景独立的 activityId 和用户类型风控逻辑。
