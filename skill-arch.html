  → userId hash 稳定分桶

  → 并行搜关怀店，3秒超时

  → healthExtraStore / healthCoupon
📝 Skill 层（Markdown · Lion 配置）
• 场景识别指引
  → 4 种场景的语气和禁忌

  → 过渡词、推荐风格、自然度要求
• 红包引出方式
  → 不复述金额，自然过渡
• 整体原则
  → 自然 > 完整，关心 > 推销

📝health_care Skill Markdown

这就是配置到 Lion skill.md.map["health_care"] 的完整内容。LLM 调用 activate_skill(skill_id="health_care") 后会拿到这份"说明书"。

与 explore skill 的对比：explore 是"主Skill + 4个子Skill"的渐进式披露结构，因为涉及多步 API 调用。健康关怀只需要一个主 Skill——不需要子 Skill，不需要 meituan_api_request，因为搜店/发券由代码层完成，Skill 只负责话术策略。

⚙️Lion 配置变更

迁移到 Skill 方案后需要在 Lion 配置中心做的变更。

1. skill.md.map 新增 key

// Lion key: skill.md.map // 新增 health_care → Skill Markdown 全文 { "weather_skill": "# 天气技能\n...", "music_skill": "# 音乐技能\n...", "explore": "# 逛逛探索\n...", // ↓ 新增 ↓ "health_care": "# 健康关怀 · 人性化主动营销\n\n你是小美..." }

2. System Prompt 注入路由规则

// Lion key: prompt.tool_call_system_prompt // 末尾追加 Skill 路由指令（Phase 1 方案 A） ## Skill 路由规则 当 health_context 被注入时（即系统检测到用户需要关怀）， 优先调用 activate_skill(skill_id="health_care") 获取关怀策略指引， 然后按指引在正常推荐后追加关怀内容。

🎯4 场景 Skill 策略详情

以下是 Skill Markdown 中各场景的具体策略，LLM 会根据 health_context 中的 scene_type 匹配对应策略。

🍔
不健康饮食提醒
scene_type: unhealthy
触发Horus persona 高油高辣占比 > 60%
搜店query = "粥 轻食 沙拉"
语气轻松建议，不说教

过渡词："对了，" "话说回来，"
示例："看你最近几天吃了不少重口味的，今天换个清淡的试试？"
禁忌：不提"不健康""垃圾食品"，不做医学判断
🌧️
天气变化关怀
scene_type: weather
触发当日降温≥8°C 或降雨
搜店query = "砂锅 热汤 粥"
语气温暖关心

过渡词："今天降温了，" "外面下雨呢，"
示例："来碗热汤暖暖，窝着喝一碗超幸福~"
禁忌：天气信息必须准确，不说错天气状况
🌙
深夜加班关怀
scene_type: latenight
触发工作日 22:00-02:00 点餐
搜店query = "粥 面 蒸饺 宵夜"
语气体贴但不煽情

过渡词："这个点还在忙呀，" "加班辛苦了，"
示例："又好吃又不太负担，暖胃又管饱~"
禁忌：周末不说"加班"，不说"熬夜伤身"
💆
情绪关怀
scene_type: mood
触发NLU 检测到消极/低落语气
搜店query = "粥 甜品 暖食"
语气温柔陪伴，不追问

过渡词："感觉你今天可能有点累，"
示例："热乎乎的喝完心情都会好一点~"
禁忌：绝对不说"你心情不好"，用"可能有点累"替代
Skill 的优势体现：以上所有策略内容都在 Markdown 里，PM 想改语气、换过渡词、调整禁忌，直接在 Lion 配置中心编辑 skill.md.map 就实时生效，不需要研发排期和发版。
